- Recovery commands run with the **current process permissions** — no privilege escalation
- The command string is logged in full — operators can audit what was executed
- In production, recovery commands SHOULD be restricted to known scripts in configured paths

### 10.2 Idempotency Guard (OPTIONAL, for agentic/retry workflows)

An Idempotency Guard prevents duplicate execution when operations are retried.

#### When to Use

- AI agent retry loops (agent fails, framework re-executes)
- Scheduled tasks that may overlap
- API endpoints that may receive duplicate requests
- Pipeline stages that may be re-run after partial failure

#### Why This Is an Extension, Not Core

Idempotency Guards are **invasive** — they change function signatures, require storage infrastructure (database columns, Redis keys), and force callers to provide idempotency keys. This breaks the core standard's principle of **additive adoption** (§1.4, principle 4). Guard, Trace, and Verify can be added to any function without changing its interface. Idempotency cannot.

#### Pattern

```python
def create_order(order_data, idempotency_key):
    # GUARD:IDEMPOTENT — check for duplicate execution
    existing = db.query("SELECT id FROM orders WHERE idempotency_key = ?", idempotency_key)
    if existing:
        log.info(f"GUARD:IDEMPOTENT:create_order | duplicate detected, key={idempotency_key}, returning existing")
        return existing[0]
    
    # ... proceed with creation
```

This appendix documents the reasoning behind major design choices. For a more detailed, beginner-friendly explanation, see [GUIDE.md](GUIDE.md).

### B.1 Why Convention Over Library?

A library creates a dependency. Dependencies get abandoned, compromised, or version-locked. A convention is forever. You can implement Bulwark AI in any language, any project, any era — because it's just `if/raise`, `log()`, and `if/raise` again. No package manager required.

### B.2 Why Three Layers Instead of One?

Input validation (Guard) catches bad data coming in. But it doesn't tell you *where* in a pipeline the data went wrong (Trace), and it doesn't confirm that your writes actually worked (Verify). Each layer addresses a different failure mode. Combining them creates a complete defensive membrane.

### B.3 Why Structured Diagnostic Messages?

Unstructured errors like `KeyError: 'pages'` require a human to read the stack trace, understand the context, and figure out the fix. Structured messages like `GUARD:process_pages | missing 'pages', available: ['result', 'metadata']` can be parsed by an AI agent to self-correct in one iteration. This is the key insight: **error messages are an API for the next debugging agent** (human or AI).

### B.4 Why Not Always Full Verification?

Because physics. A read-back query takes 1-5ms. At 10,000 operations per second, that's 10-50 seconds of additional database time per second — impossible. The standard requires verification but allows the strategy to adapt. See §3.3.5 and §6.4.

### B.5 Why Are Guards Always On But Traces Are Configurable?

Cost. A Guard (`isinstance(data, dict)`) takes 50 nanoseconds. A Trace (format string + log write) takes 5-50 microseconds. At high throughput, Trace log volume becomes significant — not the CPU cost, but the I/O and storage cost of millions of log lines per hour. Guards have no such scaling concern.

### B.6 Why Is Idempotency an Extension, Not Core?

Adoption friction. Guard, Trace, and Verify can be added to any existing function without changing its signature, its callers, or its infrastructure. Idempotency requires a new parameter, a storage mechanism, and a policy decision. It's important, but it's a different *kind* of thing — an architectural pattern, not a defensive annotation.
