### 🏥 Availability Guards (Extension)

Check whether external services are reachable **at startup**, not per-call:

```python
def check_api_available(base_url, timeout=5):
    """Availability Guard: verify the API is reachable before starting work."""
    response = requests.get(f"{base_url}/health", timeout=timeout)
    if response.status_code != 200:
        raise RuntimeError(
            f"GUARD:availability | {base_url}/health returned "
            f"status {response.status_code}, expected 200"
        )

# Use at startup:
check_api_available("https://api.example.com")
```

Availability Guards are an extension — useful in practice, but not required for Bulwark compliance.

#### 🔄 Self-Healing Recovery (NEW in v0.2.0)

When a dependency is down and you have a script that fixes it, why not run it automatically?

```python
from bulwark import guard_available

guard_available(
    check=lambda: requests.get("http://localhost:5432/health", timeout=3).status_code < 500,
    function_name="app_startup",
    dependency="PostgreSQL",
    recovery_command="docker compose up -d postgres",  # ← runs automatically if DB is down
    max_retries=3,
    retry_delay=5.0,
)
# If DB is up → passes instantly
# If DB is down → runs recovery script → waits → re-checks → succeeds or fails loudly
```

Recovery is optional, bounded (never retries forever), and fully logged. No `recovery_command` = original behavior unchanged. See the [Guide](GUIDE.md#53-recovery-self-healing-availability-guards) for the full pattern.

### Recovery (if configured)
- [ ] Recovery command is explicit (caller-provided, not inferred)
- [ ] Recovery is bounded (`max_retries` enforced, not indefinite)
- [ ] Recovery uses exponential backoff (`retry_delay` × `backoff_multiplier`)
- [ ] Every recovery attempt is logged with: attempt number, command, exit code, elapsed time
- [ ] Recovery does NOT escalate privileges (no `sudo` injection)
- [ ] Recovery failure includes full context: dependency name, all attempts, total time, last exit code
- [ ] Omitting `recovery_command` produces identical behavior to detection-only (backward compatible)

## Configuration & Runtime Modes

### Mode Selection
- [ ] Runtime mode is set via `BULWARK_MODE` environment variable or code-level configuration
- [ ] Mode defaults to `development` when not explicitly set
- [ ] Three modes are supported: `development`, `staging`, `production`

### Mode Behavior Matrix
- [ ] **Guards**: Always active in ALL modes (non-negotiable, O(1) cost)
- [ ] **Traces**: Verbosity adapts to mode:
  - Development: `DEBUG` (all traces visible)
  - Staging: `INFO` (key boundaries visible)
  - Production: `WARNING` (only anomalies visible)
- [ ] **Verifies**: Strategy adapts to mode:
  - Development: `full` (every operation verified)
  - Staging: `sampled` (configurable sample rate)
  - Production: `adaptive` (critical=full, routine=sampled)

### Configuration Mechanism
- [ ] At least one configuration method is supported (env var, code-level, or config file)
- [ ] Configuration is documented for the deployment environment
- [ ] `critical=True` flag is applied to high-value verify operations (financial, auth, data integrity)
