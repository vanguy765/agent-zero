# Construction Voice Platform (Scaffold)

> **Current Status**: 🚧 Clean Scaffold / Starter Kit
> This repository is a configured foundation for building a high-performance, distributed application suite. It is not yet a feature-complete product.

## What is this?

This is a **Modern Monorepo Architecture** designed for teams building a "Collection of Apps" that share a single data model and "Brain".

It is pre-configured to build:
1.  **Admin Dashboards**: High-density office interfaces (SolidJS + Vite).
2.  **Field Companion Apps**: Offline-capable mobile PWAs for workers (SolidJS).
3.  **Voice-Enabled APIs**: A backend optimized for Audio processing, LLM extraction, and Realtime data (Hono).
4.  **Native Mobile Apps**: Ready for Flutter integration via OpenAPI generation.

## Why this stack?

*   **Type Safety**: If you change the database, the Mobile App break at *compile time*, not in production.
*   **Speed**: Uses `pnpm` and `Turborepo` for instant builds.
*   **Reactivity**: Uses `SolidJS` for raw performance (no Virtual DOM).
*   **Unified Logic**: Shared `packages/schemas` ensure the API and Frontend always speak the same language.

## Directory Structure

```text
/
├── apps/
│   ├── api/             # Hono Backend (The Brain)
│   │                    #  - Exposes OpenAPI / Swagger
│   │                    #  - Handles Voice Processing & Auth
│   ├── web-dashboard/   # SolidJS (Office Admin Admin)
│   ├── site-companion/  # SolidJS PWA (Mobile Field Worker)
│   └── mobile-flutter/  # (Placeholder) Future Native App
├── packages/
│   ├── schemas/         # THE SOURCE OF TRUTH (Zod definitions)
│   ├── db/              # Database Types (Auto-generated from Supabase)
│   └── ui/              # Shared Design System
```

## Where to Begin Development?

**1. The Mental Model: "Schema First"**
Don't start by building a button. Start by defining the **Data**.
*   **Step 1**: Go to `packages/schemas/src`. Define what a "Task" or "Material" looks like.
*   **Step 2**: Go to `apps/api/src/routes`. Create an endpoint that accepts that Schema.
*   **Step 3**: Go to `apps/web`. Build the UI that sends that Data.

**2. Setup**
```bash
# 1. Install dependencies
pnpm install

# 2. Start the Database (Docker)
npx supabase start

### 3. Start Development

*   *(See [docs/ports.md](./docs/ports.md) for full list)*

**3. Useful Commands**
*   `pnpm db:types`: Run this after you add a table to Supabase. It updates the TypeScript definitions in configured packages.
*   `http://localhost:3000/ui`: View the Swagger API documentation.

## 🔧 Troubleshooting & Known Issues

### Supabase Local Config
If `supabase start` fails with parsing errors, ensure `supabase/config.toml` does **not** contain deprecated keys for your CLI version:
*   `deno_version`: Comment this out if using standard CLI.
*   `health_timeout`: Comment this out if causing parsing errors.
*   `major_version`: Must match your valid data directory (default is **17**, do not downgrade to 15 unless starting fresh).

### Realtime Updates Not Appearing
If the Dashboard connects (`SUBSCRIBED`) but receives no data:
1.  **RLS Policies**: Realtime respects RLS. If you are "Anonymous", strict policies (`uid=user_id`) will hide data. Use a permissive policy for testing or ensure `Authorization` headers are sent.
2.  **Versioning**: Ensure the table is in the `supabase_realtime` publication (`alter publication supabase_realtime add table ...`).

### Login/Seed Issues
`supabase db reset` might fail on complex seed files. Use the API scripts in `apps/api/scripts/` to reliably seed data:
*   `create_user.ts`: Creates the default 'Alice' user.

# Document Parser Service (FastAPI)

This service provides a GPU-accelerated document parsing API using **Detectron2**, **LayoutParser**, and **PyMuPDF**. It extracts text, renders pages to images (uploaded to Supabase), and identifies structured data like tables and landmarks (floor plans).

## Getting Started

The service is configured to run automatically as a FastAPI server on port **5000** when you start the container.

### 1. Configure Environment
Ensure your `docker-compose.yml` has the correct `SUPABASE_URL` and `SUPABASE_SERVICE_KEY`.

### 2. Start the Service
```bash
cd doc-parser
docker compose up -d
```

## API Endpoints

### `GET /`
Health check endpoint. Returns GPU availability and Supabase connection status.

### `POST /parse`
Main parsing endpoint.
- **Form Data**:
    - `project_name`: The name of the project (used for Supabase storage paths).
    - `file`: The PDF or image file to parse.

**Returns**: JSON containing page metadata, text content, image URLs, and detected landmarks.

## Integration Examples

### Node.js (using fetch)

Instead of executing commands directly, you can now call the service via HTTP. This is much faster and more robust.

```javascript
const fs = require('fs');
const FormData = require('form-data');

async function parseDocument(projectId, filePath) {
    const form = new FormData();
    form.append('project_name', projectId);
    form.append('file', fs.createReadStream(filePath));

    try {
        const response = await fetch('http://localhost:5000/parse', {
            method: 'POST',
            body: form,
            headers: form.getHeaders()
        });

        if (!response.ok) {
            throw new Error(`Error: ${response.statusText}`);
        }

        const data = await response.json();
        console.log('Parsed Data:', data);
        return data;
    } catch (error) {
        console.error('Parsing failed:', error);
    }
}
```

## Cross-Container Networking

If you are trying to reach this service from another Docker container (e.g., **Agent Zero** or another microservice), you have two main options:

### 1. Using `host.docker.internal` (Recommended for Windows/Mac)
If your other container is also running on Docker Desktop, it can reach the host machine using a special DNS name. Use this address in your other container:
- **Address**: `http://host.docker.internal:5000/parse`

### 2. Using a Shared Docker Network (Cleaner)
You can create a shared network so containers can talk to each other using their service names.

**Step 1: Create a network (if you don't have one)**:
```bash
docker network create agent-zero-network
```

**Step 2: Update `docker-compose.yml`**:
Add the network to your `doc-parser` service:
```yaml
services:
  doc-parser:
    ...
    networks:
      - agent-zero-network

networks:
  agent-zero-network:
    external: true
```

**Step 3: Access from another container**:
Once the other container is also joined to `agent-zero-network`, it can reach this service using the service name as the hostname:
- **Address**: `http://doc-parser:5000/parse`

## Development

The container is set up with `--reload`, so changes to `server.py` will automatically restart the FastAPI server inside the container.
