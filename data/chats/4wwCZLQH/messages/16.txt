# Understanding the Monorepo & Supabase

You asked some great questions about the folder structure. Here is exactly what is happening under the hood.

## 1. The Mystery Folder: `node_modules/@repo/tsconfig`

You noticed this path:
`packages/schemas/node_modules/@repo/tsconfig`

### What is it?
This is a **Symlink** (Symbolic Link). Think of it as a "Portal" or "Shortcut".

### Why is it there?
In `packages/schemas/package.json`, we have this dependency:
```json
"devDependencies": {
  "@repo/tsconfig": "workspace:*"
}
```
The `workspace:*` tells pnpm: "Don't go to the internet (npm registry) to find this. Look in my modules folder."

So, pnpm creates a shortcut:
*   **From**: `packages/schemas/node_modules/@repo/tsconfig`
*   **To**: `packages/tsconfig` (The actual folder where your JSON lives)

**Benefit**: If you edit `packages/tsconfig/base.json`, the changes apply *instantly* to `packages/schemas` because they are literally looking at the same file on your disk.

---

## 2. "Installing" Supabase

Supabase is unique. It is not just a library; it is a full **Infrastructure Stack** (Postgres Database, Auth Server, Realtime Server, Storage Server).

### How to Install it
You don't "npm install" the database itself. You run it using **Docker**.

1.  **Initialize Config** (I am doing this for you now):
    ```bash
    npx supabase init
    ```
    This creates a `/supabase` folder with a `config.toml`.

2.  **Start the Engine**:
    ```bash
    npx supabase start
    ```
    This command:
    *   Downloads Docker images for Postgres, Auth, etc.
    *   Spins them up on your computer.
    *   Gives you an **API URL** and **Anon Key**.

---

## 3. The Relationship: `packages/db` vs. "Supabase"

It is easy to confuse the **Tool** with the **Code**.

| Component | What it is | Where it lives |
| :--- | :--- | :--- |
| **Supabase Service** | The Engine. Holds your actual data (Users, Tasks). | **Docker Deployment** (or Cloud) |

# Construction Voice Platform: Comprehensive Developer Handbook

> [!NOTE]
> This document serves as the master architectural guide for the Construction Voice Platform. It is designed to be a living document that teaches *why* we made these technical choices, not just *how* to use them.

## 1. The Philosophy: Lean, Type-Safe, and Distributed

We are building a **Collection of Apps**, not a monolithic disaster. A construction company needs a Dashboard for the office, a Mobile App for the site, and potentially a Kiosk for the warehouse. These apps are different, but they share the same DNA: the **Data Model**.

### Core Principles

1.  **Monorepo (Single Source of Truth)**: We use a monorepo so that when we update a data model (e.g., "A Brick now weighs 2kg"), that change propagates instantly to the Backend, the Dashboard, and the Mobile App. No desyncs. No "it works on my machine."
2.  **Schema-First Development**: We do not write code until we define the Shape of the data. We use **Zod** as our contract. If it's not in the Zod schema, it doesn't exist.
3.  **Visual Grounding**: Voice without context is useless. When a user says "Delete it," our system must know what "it" is by inspecting the user's screen state.
4.  **Local-First / Edge-Ready**: We develop locally with Supabase and Hono, but our architecture is designed to run on the Edge (Cloudflare Workers) for low latency.

---

## 2. The High-Performance Stack

This stack is chosen for speed (both runtime and development velocity) and type safety.

| Component | Technology | Why? |
| :--- | :--- | :--- |
| **Monorepo Manager** | **Turborepo** + **pnpm** | Fast builds, efficient caching, and top-tier workspace support. |
| **Frontend** | **SolidJS** | True reactivity (signals) without the virtual DOM overhead. Performance defaults. |
| **Backend** | **Hono** | Ultra-lightweight web standard compliant framework. Runs anywhere (Node, Cloudflare, Deno). |
| **API Communication** | **Hono RPC** | End-to-end type safety without generating SDKs. The frontend imports the backend's *types*. |
| **Database & Auth** | **Supabase** | Postgres power with easy Auth, Realtime subscriptions, and Storage. |
| **Validation** | **Zod** | Runtime validation + Static Type inference. The backbone of our system. |
| **Voice Processing** | **OpenAI Whisper** + **LLM** | High-fidelity transcription followed by structured data extraction. |

---

## 3. Project Architecture

The directory structure is critical. It separates "Runnable Apps" from "Shared Logic".

```text
/construction-voice-platform
├── /apps                     # EXECUTABLE APPLICATIONS
│   ├── /web-dashboard       # SolidJS (Admin/Office view)
│   ├── /site-companion      # SolidJS PWA (Mobile view for workers)
│   └── /api                 # Hono server (The Brain / Voice Orchestrator)
├── /packages                 # SHARED LOGIC (The "Glue")
│   ├── /db                  # Supabase generated types + DB utilities
│   ├── /schemas             # THE CONTRACT (Shared Zod schemas)
│   ├── /ui                  # Shared SolidJS components (Buttons, Layouts)
│   └── /tsconfig            # Shared TS configurations
├── /supabase                 # Database config
│   ├── config.toml
│   └── /migrations          # SQL files (Single source of truth for DB structure)
├── /docs                     # Documentation (ADRs, Guides)
├── turbo.json                # Build orchestration config
├── package.json              # Root dependencies
└── README.md
```

> [!TIP]
> **Junior Dev Tip**: Never import from `apps/` into `packages/`. Logic flows *down* from packages to apps. Packages are stateless utilities; Apps are stateful implementations.

# Port Allocation Contract

To avoid conflicts during development, we strictly assign specific ports to each application in the monorepo.

## Assigned Ports

| Port      | Application         | Path                  | Notes                                                                 |
| :-------- | :------------------ | :-------------------- | :-------------------------------------------------------------------- |
| **3000**  | **API**             | `apps/api`            | The Backend (Hono) + A2A Protocol (`/a2a`, `/.well-known/agent.json`) |
| **3001**  | **Web Dashboard**   | `apps/web-dashboard`  | Admin/Office View (Legacy/Reference)                                  |
| **3002**  | **Material Admin**  | `apps/material-admin` | Material & Taxonomy Manager                                           |
| **3003**  | **Site Companion**  | `apps/site-companion` | Mobile Companion App (Voice Log)                                      |
| **3004**  | **Work Orders**     | `apps/work-orders`    | Work Orders Management                                                |
| **5000**  | **Doc Parser**      | `doc-parser`          | Python FastAPI Service for LayoutParser                               |
| **8082**  | **NocoDB**          | -                     | No-code Database Interface (Internal Tools)                           |
| **54321** | **Supabase**        | -                     | Local Database (Postgres)                                             |
| **54323** | **Supabase Studio** | -                     | Local Dashboard UI                                                    |
| **5080**  | **Agent Zero**      | `agent-zero`          | Agent Zero                                                            |

## Rules

1.  **Strict Ports**: In `vite.config.ts`, always set `server.port` to the assigned value.
2.  **No Random Ports**: Do not let Vite pick a random port. This breaks CORS configurations.
3.  **Check this doc**: Before creating a new app, claim the next available port here.

## Troubleshooting

### Windows "Zombie" Processes

If you encounter `EADDRINUSE` or random port assignments (e.g., 3009 instead of 3002), it is likely that a previous Node process is still running in the background.

**Fix**: Run this command in PowerShell to Force-Kill all Node processes:

```powershell
taskkill /F /IM node.exe
```

---

## Docker / Compose port mapping (local development)

The repository provides a `docker-compose.yml` (root of `agent-zero`) that maps the service ports for the single "agent-zero" service. By default the compose file maps:

- Container port 80 -> Host 80
- Container port 9000 -> Host 9000
- Container port 22 -> Host 22

If you run multiple services from the monorepo in parallel, ensure these host ports are free or modify the `docker-compose.yml` port mappings to avoid conflicts.

Tip: use `docker compose ps` and `docker compose logs -f <service>` to inspect running services and assigned ports.
