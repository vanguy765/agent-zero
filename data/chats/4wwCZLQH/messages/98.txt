### 8.5 Tenant Portal Access Flow

```
1. Admin configures portal_table_config for an organization
   (sets tables as hidden/readonly/editable, column visibility)
         ↓
2. Tenant logs in via Supabase Auth (email/password or magic link)
         ↓
3. Dashboard fetches portal_table_config (RLS filters by org)
         ↓
4. Tenant clicks a table → TableViewPage queries supabase.from(tableName)
   (RLS automatically filters to tenant's organization data)
         ↓
5. For editable tables: click cell → inline edit → supabase.update()
   (RLS prevents cross-tenant writes)
```

## 9. Strategic Direction & Recommendations

### 9.1 Immediate Next Steps

1. **Implement Timesheet Tables**: The schema is designed (`docs/timesheet/schema_design.md`). Create the migration for `timesheets`, `work_logs`, and `site_items` tables.

2. **Voice-to-Timesheet Pipeline**: Connect the existing voice transcription to the timesheet data model. This is the platform's killer feature — workers speak their hours and the system creates structured time entries.

3. **Site Companion Buildout**: The PWA scaffold exists but needs the full work order execution UI (room checklists, step completion, photo capture).

4. **RLS Policy Hardening**: Current policies are permissive for development. Production requires proper `organization_id` scoping and role-based access.
