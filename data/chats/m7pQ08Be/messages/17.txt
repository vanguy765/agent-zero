## 5. Runtime Modes: Not Everything Costs the Same

You might be thinking: "All these checks sound great, but won't they slow down my code?" Great question. The answer is: **it depends on the layer**, and Bulwark AI is designed with this in mind.

### The Car Analogy

Think of the three layers like safety features in a car:

🟢 **Guards = Seatbelt.** Always on. Costs nothing. You buckle up every single time you get in the car, whether you're driving to the grocery store or across the country. It takes half a second and could save your life. You would never say, "I'm just going to the store, I'll skip the seatbelt." Guards are the same — they cost nanoseconds (billionths of a second) and catch the most dangerous failures. **You never turn them off.**

🟡 **Traces = Dashcam.** Always recording, but you only review the footage after an incident. You can set the video quality to save storage — maybe 4K during a road trip through unfamiliar territory, but 720p for your daily commute. The camera is always there, always running, but how much detail it captures can be adjusted. Traces work the same way — the code is always present, but you can turn the verbosity up or down depending on the environment.

🔵 **Verifies = Vehicle inspection.** How thorough the inspection is depends on the context. Going on a cross-country trip? Full bumper-to-bumper inspection. Running to the grocery store? Quick check — tires look fine, gas tank isn't empty, good to go. Managing a taxi fleet? Spot-check a few cars each day; you can't inspect every car before every ride. Verifies work the same way — full verification for critical operations, lighter strategies for high-throughput scenarios.

### What Changes Across Environments

| Layer | Development | Staging / CI | Production |
|-------|------------|-------------|------------|
| **Guard** | ✅ Active — crash with diagnostic | ✅ Active — crash with diagnostic | ✅ Active — crash with diagnostic |
| **Trace** | ✅ Verbose — full shape at every boundary | ✅ Verbose — full shape for test visibility | ⚙️ Configurable — summary or silent, but code stays |
| **Verify** | ✅ Full — read-back every operation | ✅ Full — catch issues before production | ⚙️ Adapts — full, batch, or sampled based on throughput |

### Guards: Never Turned Off

This is worth repeating: **Guards are never turned off.** Not in development, not in staging, not in production. Here's why:

- A Guard that checks `isinstance(data, dict)` takes about 50 nanoseconds.
- The function it protects (a database query, an API call) takes 5-500 milliseconds.
- That means the Guard adds less than 0.001% overhead.
- The failures Guards catch — wrong data shape from an API, missing keys from a database result — happen *in production*, when real services change their response formats.

Disabling Guards in production would be like removing your seatbelt on the highway because "the road looks clear." The danger is exactly where you can't predict it.

### Traces: Always Present, Volume Adjustable

Trace *code* is always in your source files. But the *output* (the log lines) is controlled by your logging configuration:

- **Development:** Set log level to `DEBUG`. You see full shape descriptions at every boundary. Great for understanding data flow.
- **Staging:** Set log level to `DEBUG` or `INFO`. Full visibility for catching issues in testing.
- **Production:** Set log level to `INFO` or `WARNING`. At `INFO`, you get summary traces (type and size only). At `WARNING` or above, traces are silent — but the code is still there. If a production issue occurs, you lower the log level and immediately see the data flow without deploying new code.

### Verifies: Strategy Adapts to Throughput

Verification is always required, but *how* you verify adapts to your situation:

| Scenario | Strategy | Why |
|----------|----------|-----|
| A CRUD endpoint handling 50 requests/minute | Full per-operation read-back | 50 extra queries/minute is negligible |
| A batch import of 10,000 records | One aggregate check after the batch | `SELECT COUNT(*) WHERE batch_id = ?` — one query instead of 10,000 |
| A streaming pipeline at 10,000 events/second | Verify a 1-5% sample | Full verification would double database load |
| A payment processing function | Full per-operation, always | Too critical for sampling — every transaction matters |

The key insight: **the standard requires *that* you verify, not *how* you verify.** Choose the strategy that matches your throughput and criticality.

## 6. How to Add Bulwark to Your Project

Ready to start? Here's a step-by-step guide. The most important thing to know: **you can do this one function at a time.** You don't need to refactor your entire codebase. Pick one function, add the layers, and expand from there.

### Step 1: Copy the Helper Functions

Grab the reference implementation for your language from the `reference/` directory:

- **Python:** [`reference/bulwark.py`](reference/bulwark.py)
- **TypeScript:** [`reference/bulwark.ts`](reference/bulwark.ts)

Copy the file into your project. These helpers provide `guard()`, `guard_type()`, `guard_keys()`, `trace()`, `describe_shape()`, and `verify()` — all implemented with zero external dependencies.

If you prefer, you can skip the helpers entirely and write raw `if/raise` statements. The helpers are a convenience, not a requirement.

### Step 2: Pick Your First Function

Start with a function that **reads from an external source** — an API call, a database query, or a file read. These are the most common sources of AI code failures because the data shape is unpredictable.

Good first candidates:
- A function that fetches data from a REST API
- A function that reads configuration from a file
- A function that queries a database and processes the results

### Step 3: Add Guards at the Top

At the very beginning of your function, before any business logic, add Guards that check:

1. **Presence** — Is the data `None`/`null`/`undefined`?
2. **Type** — Is it the type you expect (dict, list, string)?
3. **Shape** — Does it have the keys/fields/indices you need?
4. **Non-emptiness** — If it's a collection, does it have elements?

def process_api_data(response):
    # --- GUARD ---
    guard_type(response, dict, "process_api_data", "response")
    guard_keys(response, {"data", "status"}, "process_api_data", "response")
    guard(len(response["data"]) > 0, "process_api_data",
