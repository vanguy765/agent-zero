-- Construction Management System Schema
-- PostgreSQL DDL

-- Enable UUID extension
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Enums
CREATE TYPE project_status AS ENUM ('bidding', 'active', 'archived');
CREATE TYPE item_status AS ENUM ('pending', 'ordered', 'delivered', 'installed', 'verified');
CREATE TYPE timesheet_status AS ENUM ('draft', 'submitted', 'approved', 'rejected');
CREATE TYPE product_availability AS ENUM ('in_stock', 'order', 'discontinued');

-- 1. Projects
CREATE TABLE projects (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name TEXT NOT NULL,
    code TEXT UNIQUE NOT NULL,
    address TEXT,
    contact_info JSONB,
    status project_status DEFAULT 'bidding',
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 2. Project Definitions (JSONB Templates)
CREATE TABLE project_definitions (
    project_id UUID PRIMARY KEY REFERENCES projects(id) ON DELETE CASCADE,
    taxonomy_version TEXT,
    floorplan_definitions JSONB NOT NULL DEFAULT '{}'::jsonb,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 3. Units (Inventory)
CREATE TABLE units (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    project_id UUID REFERENCES projects(id) ON DELETE CASCADE,
    unit_number TEXT NOT NULL,
    floor_level INT,
    floorplan_key TEXT NOT NULL, -- Links to key in project_definitions
    specific_overrides JSONB DEFAULT '{}'::jsonb, -- Unit-specific exceptions
    UNIQUE(project_id, unit_number)
);

-- 4. Site Items (BOM - Derived/Exploded)
CREATE TABLE site_items (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    unit_id UUID REFERENCES units(id) ON DELETE CASCADE,
    subject TEXT NOT NULL, -- e.g., 'toilet'
    location TEXT NOT NULL, -- e.g., 'Master Bath'
    action TEXT DEFAULT 'install_new', -- 'replace', 'keep'
    
    -- Status Tracking
    status item_status DEFAULT 'pending',
    percent_complete INT DEFAULT 0 CHECK (percent_complete BETWEEN 0 AND 100),
    
    -- Estimates
    labor_estimate_hours DECIMAL(10, 2),
    
    -- Metadata
    source TEXT DEFAULT 'original_scope', -- 'original_scope' or 'change_order'
    notes TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_site_items_unit ON site_items(unit_id);
CREATE INDEX idx_site_items_status ON site_items(status);

-- 5. Suppliers
CREATE TABLE suppliers (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name TEXT NOT NULL,
    website TEXT,
    contact_info JSONB,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 6. Products (Catalog)
CREATE TABLE products (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    supplier_id UUID REFERENCES suppliers(id),
    subject_key TEXT NOT NULL, -- Matches taxonomy subject
    model_number TEXT,
    name TEXT NOT NULL,
    description TEXT,
    attributes JSONB DEFAULT '{}'::jsonb, -- Spec matching
    unit_price DECIMAL(10, 2),
    product_page_url TEXT,
    image_url TEXT,
    notes TEXT,
    availability_status product_availability DEFAULT 'in_stock',
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 7. Project Supplies (Consumables)
CREATE TABLE project_supplies (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    supplier_id UUID REFERENCES suppliers(id),
    name TEXT NOT NULL,
    description TEXT,
    product_page_url TEXT,
    image_url TEXT,
    notes TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 8. Timesheets
CREATE TABLE timesheets (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID NOT NULL, -- Assumes external auth system
    date DATE NOT NULL,
    total_hours DECIMAL(5, 2) DEFAULT 0,
    status timesheet_status DEFAULT 'draft',
    submitted_at TIMESTAMP WITH TIME ZONE,
    approved_at TIMESTAMP WITH TIME ZONE
);

-- 9. Work Logs (Time Rows)
CREATE TABLE work_logs (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    timesheet_id UUID REFERENCES timesheets(id) ON DELETE CASCADE,
    site_item_id UUID REFERENCES site_items(id), -- Link to specific BOM item
    hours DECIMAL(5, 2) NOT NULL,
    milestone_reached TEXT, -- e.g., 'Prime (30%)'
    notes TEXT,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_work_logs_item ON work_logs(site_item_id);

# Schema Design: Hybrid Relational + JSONB

## Strategy
We use **PostgreSQL** with a hybrid approach to balance performance and flexibility.
*   **Relational Columns**: Used for Foreign Keys, Indexing, Sorting, and Permissions.
*   **JSONB Columns**: Used for the deep hierarchical definition of the "Scope of Work", Attributes, and flexible metadata.

## Tables

### 1. `projects`
High-level container for a job site.
*   `id` (UUID, PK)
*   `name` (Text)
*   `code` (Text, Unique) - e.g., "PRJ-2026-001"
*   `status` (Enum: bidding, active, archived)

### 2. `project_definitions`
The "Brain" of the project. Steps away from relational rows to store the Template hierarchy.
*   `project_id` (FK)
*   `taxonomy_version` (Text)
*   **`floorplan_definitions` (JSONB)** - A dictionary of floorplans (`Plan A`, `Plan B`) and their default rooms/items.

### 3. `units`
The inventory of saleable/workable spaces.
*   `id` (UUID, PK)
*   `project_id` (FK)
*   `unit_number` (Text) - "101"
*   `floor_level` (Int)
*   `floorplan_key` (Text) - Links to `project_definitions`.
*   **`specific_overrides` (JSONB)** - Stores exceptions for *this unit only*.

### 4. `site_items` (The BOM)
Derived table. Exploded from JSON for tracking status.
*   `id` (UUID, PK)
*   `unit_id` (FK)
*   `subject` (Text) - "toilet"
*   `location` (Text) - "Master Bath"
*   `status` (Enum: pending, installed, verified)
*   `percent_complete` (Int)
*   `labor_estimate_hours` (Decimal)

### 5. `timesheets`
The container for a day's work.
*   `id` (UUID, PK)
*   `user_id` (FK)
*   `date` (Date)
*   `approval_status` (Enum)

### 6. `work_logs`
The link between Time and Scope.
*   `id` (UUID, PK)
*   `timesheet_id` (FK)
*   **`site_item_id` (FK)** - Links time to the specific BOM item.
*   `hours` (Decimal)
*   `milestone_id` (Text) - e.g., "Prime (30%)"

### 7. `suppliers` & `products`
The Catalog.
*   `suppliers`: Vendors.
*   `products`: Major items with `attributes` (JSONB) for spec matching and `image_url`.
*   `project_supplies`: Consumables list (brushes, tape).

# Timesheet Agent & System Contracts

You asked how the Timesheet Agent "knows" what to log against, and how the Timesheet App talks to the Core BOM System.

## 1. How the Agent Knows the Context

The Agent does **not** memorize the entire database. Instead, we use **Context Injection** (RAG-lite) at the moment of transcription.

### The Problem
*   User says: "I finished the toilet."
*   Database has: 500 toilets.
*   Agent needs to know: *Which* toilet?

### The Solution: "My Active Tasks" Context
When the user hits "Record" on the Timesheet App, the app acts as follows:

1.  **App Query**: The App asks the Core System: *"Give me the open BOM items for this User (assigned to Plumber) and this Location (Unit 105)."*
2.  **Core Response**: Returns a short JSON list:
    *   `id: A1`, `name: "Toilet Install"`, `status: "Pending"`
    *   `id: B2`, `name: "Sink Install"`, `status: "Pending"`
3.  **Prompt Injection**: The App sends this list to the LLM alongside the voice audio.

#### The Timesheet System Prompt
```text
You are a Time Tracking Assistant.
User Audio: "I finished the toilet in 105, took 2 hours."
Context (Available Tasks):
- [ID: A1] Toilet Install (Unit 105)
- [ID: B2] Sink Install (Unit 105)

Goal: Map the audio to a Task ID and extract hours.
Output: { "task_id": "A1", "action": "complete", "hours": 2.0 }
```

**Result**: The Agent doesn't need to know the whole Taxonomy. It only needs to pick from the *relevant short-list* provided by the API at runtime.

## 2. The System Contract (API)

The **Timesheet App** (Client) and the **Core BOM System** (Server) communicate via a strict API Contract.

### A. GET /api/v1/context/tasks
*Used by the Timesheet App to prime the Agent.*

**Request**:
*   `project_id`: UUID
*   `location`: "Unit 105" (Optional, derived from GPS or selection)
*   `trade`: "Plumbing" (Derived from User Profile)

**Response (The Contract)**:
```json
[
  {
    "id": "550e8400-e29b...", // The BOM Item ID (site_items.id)
    "label": "install toilet",
    "subject_taxonomy": "plumbing.toilet", // Helpful for the LLM to match synonyms
    "current_status": "pending",
    "milestones": ["Rough-in", "Trim-out"] // If defined in Taxonomy
  }
]
```

### B. POST /api/v1/timesheets/log
*Used by the Timesheet App to save the result.*

**Payload (The Contract)**:
```json
{
  "user_id": "uuid...",
  "logs": [
    {
      "site_item_id": "550e8400-e29b...",  // LINK TO BOM
      "hours": 2.0,
      "status_change": {
        "new_status": "installed",
        "milestone_reached": "Trim-out"
      },
      "notes": "Flange was cracked, replaced it."
    }
  ]
}
```

## 3. Handling Ambiguity (The Feedback Loop)

If the Core System returns *multiple* matching tasks (e.g., "Master Bath Toilet" vs "Guest Bath Toilet") and the user just says "Toilet":

1.  **Agent Response**: The Agent detects ambiguity based on the Context List.
2.  **Output**: `{ "error": "ambiguous", "options": ["Master", "Guest"] }`
3.  **App UI**: The App asks the user: *"Which toilet? Master or Guest?"*

This keeps the "Intelligence" grounded in the actual project data supplied by the Core BOM System.
