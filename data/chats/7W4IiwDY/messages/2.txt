# 🛡️ Bulwark AI

### Defensive Coding Standard for AI-Generated Code

**Guard · Trace · Verify**

## The Problem

AI code generation (LLMs, copilots, coding agents) produces code through **statistical pattern completion**, not runtime reasoning. The result: code that compiles, passes review, and runs — until it encounters reality.

| AI Failure Mode | What Happens |
|---|---|
| **Silent data shape mismatch** | API returns `{result: {pages: ...}}` but code reads `response["pages"]` |
| **Unverified side effects** | DB insert returns 200 but writes 0 rows — code reports "success" |
| **Confident wrong assumptions** | Assumes path is writable, service is reachable, key exists |
| **Happy-path-only logic** | No handling for empty results, missing keys, or null values |
| **Placeholder data in production** | Hardcoded `"sk-test-1234"` or `"https://example.com"` |

These aren't bugs. They're a **predictable failure class** unique to AI-generated code.

## The Solution: Three Defensive Layers

```
┌─────────────────────────────────────────────────────────────────┐
│                        YOUR FUNCTION                            │
│                                                                 │
│  ┌─── GUARD ──────────────────────────────────────────────┐     │
│  │ Validate shape, type, presence of inputs               │     │
│  │ "If this data were wrong, would code silently continue?"│     │
│  └────────────────────────────────────────────────────────┘     │
│                           │                                     │
│                    ┌──────▼──────┐                              │
│                    │  TRACE      │                              │
│                    │  Log shape  │                              │
│                    │  at phase   │                              │
│                    │  boundary   │                              │
│                    └──────┬──────┘                              │
│                           │                                     │
│                   Business Logic                                │
│                           │                                     │
│  ┌─── VERIFY ─────────────▼───────────────────────────────┐     │
│  │ Confirm side effects actually happened                  │     │
│  │ "Did the write/send/delete actually work?"              │     │
│  └────────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────────┘
```

### Layer 1: GUARD — *"Don't trust what you didn't create"*

Every function receiving external data validates inputs **before** processing.

```python
def process_pages(api_response):
    # GUARD: validate before processing
    assert isinstance(api_response, dict), \
        f"GUARD:process_pages | expected dict, got {type(api_response).__name__}"
    assert "pages" in api_response, \
        f"GUARD:process_pages | missing 'pages', available: {list(api_response.keys())}"
```

**Without Guard:** `KeyError: 'pages'` — useless.
**With Guard:** `GUARD:process_pages | missing 'pages', available: ['result', 'metadata']` — fixable in one shot.

### Layer 2: TRACE — *"Make the invisible visible"*

Log data **shape** (not content) at every pipeline phase boundary.

```python
raw = trace("after_fetch", fetch(url))       # TRACE:after_fetch | dict, keys=['result','meta'], len=2
parsed = trace("after_parse", parse(raw))     # TRACE:after_parse | list, len=47, items=dict{keys=['id','content']}
```

Traces create a **flight recorder** — when Stage 3 breaks, you see exactly what Stage 2 handed it.

### Layer 3: VERIFY — *"Trust, but verify"*

After every side effect, **confirm** it actually happened.

```python
result = db.insert_many("results", records)
# VERIFY: don't just hope — confirm
assert result.rowcount == len(records), \
    f"VERIFY:save | INSERT expected {len(records)} rows, affected {result.rowcount}"
```

**Without Verify:** Code reports success. Zero rows written. Data lost silently.
**With Verify:** `VERIFY:save | INSERT expected 47 rows, affected 0` — caught immediately.

## What's New in v0.2.0

### 🔧 Runtime Configuration

Bulwark now supports explicit configuration for runtime modes, trace verbosity, and verify strategies. Configure once at startup:

**Environment variables:**

```bash
export BULWARK_MODE=production
export BULWARK_TRACE_LEVEL=error
export BULWARK_VERIFY_STRATEGY=adaptive
export BULWARK_VERIFY_SAMPLE_RATE=0.1
```

**Or in code:**

```python
# Python
from bulwark import configure
configure(mode='production', verify_strategy='adaptive')
```

```typescript
// TypeScript
import { configure } from './bulwark';
configure({ mode: 'production', verifyStrategy: 'adaptive' });
```

### 📊 Verify Strategies

Not every verify needs a full read-back. Choose the right strategy for your scale:

| Strategy | What It Does | When to Use |
|----------|-------------|-------------|
| **full** | Verifies every operation | Low-throughput, high-stakes |
| **sampled** | Verifies a percentage of operations | High-throughput batch jobs |
| **adaptive** | Starts full, reduces as confidence grows | Production systems at scale |

> **Critical operations** (payments, auth, deletions) are **always** fully verified regardless of strategy. Mark them with the `critical` flag.

### 🏥 Availability Guards (Extension)

Check whether external services are reachable **at startup**, not per-call:

```python
def check_api_available(base_url, timeout=5):
    """Availability Guard: verify the API is reachable before starting work."""
    response = requests.get(f"{base_url}/health", timeout=timeout)
    if response.status_code != 200:
        raise RuntimeError(
            f"GUARD:availability | {base_url}/health returned "
            f"status {response.status_code}, expected 200"
        )

# Use at startup:
check_api_available("https://api.example.com")
```

Availability Guards are an extension — useful in practice, but not required for Bulwark compliance.

#### 🔄 Self-Healing Recovery (NEW in v0.2.0)

When a dependency is down and you have a script that fixes it, why not run it automatically?

```python
from bulwark import guard_available

guard_available(
    check=lambda: requests.get("http://localhost:5432/health", timeout=3).status_code < 500,
    function_name="app_startup",
    dependency="PostgreSQL",
    recovery_command="docker compose up -d postgres",  # ← runs automatically if DB is down
    max_retries=3,
    retry_delay=5.0,
)
# If DB is up → passes instantly
# If DB is down → runs recovery script → waits → re-checks → succeeds or fails loudly
```

Recovery is optional, bounded (never retries forever), and fully logged. No `recovery_command` = original behavior unchanged. See the [Guide](GUIDE.md#53-recovery-self-healing-availability-guards) for the full pattern.

### 🗂️ Service Manifest Guards (NEW in v0.2.0)

For multi-service systems, define health checks, recovery, and severity in a declarative JSON manifest:

```json
{
  "services": {
    "database": {
      "health_check": { "command": "pg_isready -h localhost" },
      "recovery": { "command": "docker compose up -d postgres", "max_retries": 3 },
      "severity": "critical"
    }
  }
}
```

`guard_service()` reads the manifest and handles health checks, recovery orchestration, and severity-driven behavior automatically. See [SPEC.md §10.2](SPEC.md) for the full schema.

## Decision Rules

Three questions. Apply to every function.

| Question | If YES |
|----------|--------|
| *"If this data were wrong, would the code silently continue?"* | Add a **GUARD** |
| *"Does data flow between processing phases here?"* | Add a **TRACE** |
| *"Does this function produce side effects?"* | Add a **VERIFY** |

## Key Principles

### 🚫 Zero Dependencies
Bulwark AI is a **convention**, not a library. Implement it with your language's built-in `assert`, `throw`, and `console.log`. No `pip install`. No `npm install`. Copy the pattern, not a package.

### 🌐 Language Agnostic
The standard works in any language. Reference implementations provided for Python and TypeScript. The patterns translate directly to Go, Rust, Java, Bash, or anything else.

### 🤖 AI-Agent Optimized
Diagnostic messages are structured so AI agents can **read their own errors and self-correct**:
```
GUARD:process_pages | missing 'pages', available: ['result', 'metadata']
```
An AI agent reads this and immediately knows: use `response['result']['pages']`.

## Quick Start

### Option 1: Copy the Reference Implementation

```bash
# Python — copy into your project
curl -o bulwark.py https://raw.githubusercontent.com/bulwark-ai/bulwark-ai/main/reference/bulwark.py

# TypeScript — copy into your project
curl -o bulwark.ts https://raw.githubusercontent.com/bulwark-ai/bulwark-ai/main/reference/bulwark.ts
```

```python
from bulwark import configure, guard, guard_keys, trace, verify, verify_file_written

# Configure at startup
configure(mode='development')

def process_and_save(api_response, output_path):
    # GUARD
    guard_keys(api_response, {"data", "metadata"}, "process_and_save", "api_response")
    
    records = transform(api_response["data"])
    records = trace("after_transform", records)  # TRACE
    
    write_json(output_path, records)
    verify_file_written(output_path, "process_and_save")  # VERIFY
```

## Diagnostic Message Format

All messages follow: `<LAYER>:<function_name> | <details>`

```
GUARD:process_pages | expected dict with key 'pages', got dict with keys: ['result', 'metadata']
TRACE:after_parse   | list, len=47, items=dict{keys=['page','content']}
VERIFY:save_records | INSERT expected 47 rows, affected 0 — FAIL
```

Structured enough for machines. Readable enough for humans.

## FAQ

**"Isn't this just input validation?"**
No. Guard is input validation. Trace is pipeline observability. Verify is side-effect confirmation. Most validation libraries don't address unverified writes or data flow visibility.

**"Won't this slow down my code?"**
Guards are O(1) type checks — nanoseconds. Traces log a shape string — microseconds. Verifies may add one lightweight query. With v0.2.0's verify strategies, you can sample or adapt verification for high-throughput systems. Unmeasurable for 99% of applications.

**"Should I use this for human-written code too?"**
Yes. But it's *designed* for AI failure modes and *optimized* for AI agent consumption.

**"Can I use pydantic/zod instead of raw assertions?"**
Absolutely. The standard defines *what* to check and *how to report failures*, not which tool to use. Validation libraries can implement the Guard layer. The zero-dep reference is always available as fallback.

**"What's the difference between runtime modes and verify strategies?"**
Runtime modes (`development`, `staging`, `production`) control the overall behavior — what gets logged, how verbose errors are. Verify strategies (`full`, `sampled`, `adaptive`) control specifically how the Verify layer operates at scale. See the [Guide](GUIDE.md#51-configuring-bulwark-ai) for the full explanation.
